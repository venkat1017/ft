Merging &
Serving Adapters

⏱ 65 min🚀 Deployment⚡ vLLM + Ollama

LoRA adapters are small (~80MB) but inference requires loading both the base model and adapter at runtime. Merging folds the adapter permanently into the base model weights - zero inference overhead, full compatibility with vLLM and Ollama. We also cover quantizing merged models for production deployment.

Merging LoRA Adapters

Merging replaces W + BA with a single matrix W' = W + BA, computed once. After merging, there is no adapter - it's just a regular model. This enables full compatibility with any inference framework and eliminates the overhead of loading and applying the adapter at runtime.

merge_adapter.py - merge LoRA into base model
PRODUCTION CODE
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

BASE_MODEL = "meta-llama/Meta-Llama-3-8B-model-Instruct"
ADAPTER_PATH = "./llama3-finetuned"  # where SFTTrainer saved adapters

# Load base model in FP16 (NOT quantized - we need full precision for merging)
base_model = AutoModelForCausalLM.from_pretrained(
    BASE_MODEL,
    torch_dtype=torch.float16,
    device_map="auto",
)

# Load adapter on top
model = PeftModel.from_pretrained(base_model, ADAPTER_PATH)

# Merge adapters into base weights (W' = W + B@A)
model = model.merge_and_unload()
# Result: regular model with no adapter structure, ready for any inference framework

# Save merged model
model.save_pretrained("./llama3-merged", safe_serialization=True)
tokenizer = AutoTokenizer.from_pretrained(BASE_MODEL)
tokenizer.save_pretrained("./llama3-merged")
print("Merged model saved - ready for vLLM, Ollama, or HF Hub")

# Push to HuggingFace Hub for easy deployment
model.push_to_hub("yourusername/llama3-8b-finetuned", private=True)
tokenizer.push_to_hub("yourusername/llama3-8b-finetuned")

Serving with vLLM (Production)

serve_vllm.sh - OpenAI-compatible API endpoint
PRODUCTION
# Install vLLM (fastest open-source LLM serving engine)
pip install vllm

# Serve your merged model as an OpenAI-compatible API
# This gives you /v1/chat/completions, /v1/completions endpoints
python -m vllm.entrypoints.openai.api_server     --model ./llama3-merged     --dtype bfloat16     --max-model-len 8192     --gpu-memory-utilization 0.9     --port 8000

# Now query it like OpenAI:
curl http://localhost:8000/v1/chat/completions     -H "Content-Type: application/json"     -d '{
      "model": "llama3-merged",
      "messages": [{"role": "user", "content": "Hello!"}],
      "temperature": 0.7
    }'
ollama_deploy.sh - run locally with Ollama
LOCAL
# Convert to GGUF format (required for Ollama)
pip install llama.cpp  # or use llama-cpp-python
python -m llama_cpp.convert_hf_to_gguf ./llama3-merged     --outfile llama3-finetuned.gguf     --outtype q4_k_m  # 4-bit quantized GGUF

# Create Modelfile for Ollama
cat > Modelfile << EOF
FROM ./llama3-finetuned.gguf
SYSTEM "You are a helpful assistant trained for [your task]."
PARAMETER temperature 0.7
PARAMETER num_ctx 4096
EOF

# Import and run
ollama create mymodel -f Modelfile
ollama run mymodel "Hello!"

# Now accessible via Ollama API or Open WebUI

What You Now Know

  • merge_and_unload(): folds B@A permanently into W, zero inference overhead
  • Must use FP16 base (not quantized) for merging - quantized can't accumulate
  • Push to HuggingFace Hub for versioning and easy sharing
  • vLLM: OpenAI-compatible API, 3-10x faster than HF generate()
  • Ollama: GGUF format for local deployment with no GPU required
