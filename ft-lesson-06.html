⏱ 60 min📈 Loss Curves🔍 Common Failures

A fine-tuning run looks healthy until it doesn't. Loss curves tell a story - but only if you know how to read them. This lesson covers the seven failure modes every engineer encounters, how to diagnose them from training logs alone, and exactly what to change.

Reading Loss Curves

Seven training patterns and what they mean
✓ Healthy Training
Train loss: smooth downward curve. Val loss: tracks train, maybe slightly higher. Gap stays stable. Both plateau around same value.
⚠ Overfitting
Train loss: still falling. Val loss: rising or plateaued much higher. Cause: too many epochs, too high rank, too little data. Fix: early stop, reduce r, add dropout.
⚠ Loss Spike Then Recovery
Sudden jump then returns to trend. Usually one bad batch. Usually harmless. If repeated: reduce LR or check for corrupted examples.
✗ Loss Diverges (NaN)
Loss goes to NaN or infinity. Causes: LR too high, bad data with empty sequences, forgot prepare_model_for_kbit_training(). Fix: halve LR first.
⚠ Loss Doesn't Fall
Stuck near initial value (e.g., ~2.5 for 5K vocab model). Causes: wrong chat template, no loss masking, LR too low, frozen adapters. Check trainable params.
✗ Loss Falls Too Fast
Drops to near 0 in first epoch. Overfitting severely. Cause: tiny dataset, very high LR. Result: model memorizes examples, can't generalize. Rebuild dataset.

Diagnostic Checklist

diagnostics.py - run these checks before and during training
DEBUGGING
# ── Check 1: Are adapters actually trainable? ─────────────────
model.print_trainable_parameters()
# MUST show: trainable params: X || all params: Y || trainable: 0.1-5%
# If trainable=0%: you forgot get_peft_model() or model is fully frozen

# ── Check 2: Is the chat template correct? ────────────────────
sample_ids = next(iter(train_loader))["input_ids"][0]
print(tokenizer.decode(sample_ids[:100]))
# Should look like:
# <|begin_of_text|><|start_header_id|>system<|end_header_id|>...
# If it looks like: "{'role': 'user', 'content': '..."
# → your dataset has the wrong format (dict not formatted yet)

# ── Check 3: Is loss masking working? ────────────────────────
sample_labels = next(iter(train_loader))["labels"][0]
masked_pct = (sample_labels == -100).float().mean().item()
print(f"Masked tokens: {masked_pct:.1%}")
# Should be 30-70% (user/system tokens masked)
# If 0%: loss masking is off - will train on user turns too
# If 99%+: almost everything masked - check chat template

# ── Check 4: VRAM usage ──────────────────────────────────────
import torch
print(f"VRAM: {torch.cuda.memory_allocated()/1e9:.1f}GB / {torch.cuda.get_device_properties(0).total_memory/1e9:.0f}GB")
# For T4 (15GB): want to use 12-14GB (some headroom for spikes)
# If OOM: reduce per_device_batch_size, increase gradient_accumulation_steps

# ── Check 5: Gradient norms ───────────────────────────────────
# Add to training loop or use wandb to track grad_norm
# Healthy: 0.01 - 1.0
# Too high (>5): LR too high or exploding gradient
# Too low (<0.001): LR too low or vanishing gradient

What You Now Know

  • Six loss curve patterns: healthy, overfitting, spike, diverge, plateau, too-fast
  • First debug step: print_trainable_parameters() and decode a training sample
  • Check masked_pct: should be 30-70% of tokens masked to -100
  • OOM: reduce max_seq_length first (biggest impact), then batch size
  • packing=True: pack multiple short examples into one sequence, 2x throughput
