Why Fine-Tuning?
Pre-Train vs Adapt

⏱ 45 min🧠 Intuition💡 Theory-First

A base LLM knows almost everything - but it answers like an internet document, not an assistant. Fine-tuning is the process of taking that raw capability and shaping behavior without re-learning facts. Understanding the difference between pre-training, instruction tuning, and RLHF is the first critical mental model.

The Three Phases of Making an LLM

Every production LLM you interact with went through at least two, usually all three, of these phases. They have completely different goals, costs, and datasets.

Phase 1: Pre-Training
Next-token prediction on trillions of web tokens. Learns world knowledge, language, reasoning. Costs $1M–$100M. Done once by labs.
Data: CommonCrawl, books, code
Goal: compress world knowledge
Result: "base model"
Phase 2: Instruction Tuning
Fine-tune on (instruction, response) pairs. Teaches the model to follow directions, answer helpfully, use chat format. Costs $100–$10K.
Data: Alpaca, ShareGPT, custom
Goal: shape behavior/format
Result: "instruct model"
Phase 3: RLHF / DPO
Human feedback or preference data aligns values: helpful, harmless, honest. Often uses PPO or DPO. Makes models safe and pleasant.
Data: preference pairs
Goal: align with human values
Result: "chat model"
Key Insight

Base models know the same facts as instruct models. The difference is purely behavioral. When you fine-tune, you are NOT teaching new knowledge - you are reshaping how that knowledge gets expressed. This is why fine-tuning needs very little data: you are changing style and format, not filling an empty brain.

When Fine-Tuning Beats Prompting

Before spending GPU time, ask: could a really good system prompt solve this? Prompting is free and instant. Fine-tuning takes hours and money. But fine-tuning wins decisively in certain cases:

Full Fine-Tune vs Parameter-Efficient Methods

Fine-tuning ALL parameters of a 7B model requires ~112GB VRAM (weights + gradients + optimizer states). Free GPUs give you 15GB. This is why the course focuses on QLoRA - Quantized Low-Rank Adaptation - which fine-tunes tiny adapter layers on top of a frozen quantized model. Same quality, 10× less memory.

QLoRA (this course)
Freeze base model in 4-bit. Train 1-5% of params as small matrices. VRAM: 10-16GB. Works on free Colab T4. Quality: 95%+ of full fine-tune.
Full Fine-Tune
Update all parameters. VRAM: 80-160GB for 7B model. Needs 8×A100 cluster. Quality: 100% ceiling. Cost: $100-$1000 per run.
LoRA (no quantization)
Train adapters on FP16 model. VRAM: 20-40GB. Faster training than QLoRA. Needs A100 or 2×3090.
Prompt Engineering
No training. But long prompts hit context limits, add latency, expose system prompts, and can't change core behavior.

The Mental Model: Adapter = Delta

LoRA trains weight updates, not weights. For any weight matrix W, instead of updating W directly, LoRA learns a small delta: W' = W + BA where B and A are thin matrices. If W is (4096, 4096), B is (4096, 16) and A is (16, 4096). That's 16M vs 131M parameters - 8x fewer. The "rank" (16 here) controls this tradeoff.

lora_intuition.py
INTUITION
import torch
import torch.nn as nn

# Frozen base model weight (e.g., attention projection in Llama-7B)
W_frozen = torch.randn(4096, 4096)  # 16.8M params - frozen, no grad
W_frozen.requires_grad = False

# LoRA adapters (trainable) - rank r=16
r = 16
A = nn.Linear(4096, r,    bias=False)  # 65,536 params
B = nn.Linear(r,    4096, bias=False)  # 65,536 params
# Total trainable: 131K vs 16.8M = 0.78% of original!

nn.init.normal_(A.weight, std=0.02)
nn.init.zeros_(B.weight)  # B init to 0 so LoRA starts as identity

def lora_forward(x, alpha=16):
    # Base path (frozen) + adapter path (trainable)
    base    = x @ W_frozen.T                   # original computation
    adapter = B(A(x)) * (alpha / r)           # low-rank update, scaled
    return base + adapter                      # W_frozen + B@A is the new W

# alpha/r scaling: keeps adapter contribution stable as r changes.
# alpha=16, r=16 → scale=1.0 (no scaling)
# alpha=16, r=8  → scale=2.0 (stronger adapter per param)
# Typical: set alpha = r (or 2*r for stronger adaptation)

What You Now Know

  • Three phases: pre-training (knowledge), instruction tuning (behavior), RLHF (values)
  • Fine-tuning changes HOW knowledge is expressed, not what the model knows
  • QLoRA: 4-bit frozen base + tiny trainable adapters - runs on free T4 GPU
  • LoRA equation: W' = W + BA where B,A are low-rank matrices
  • B initialized to zero so training starts from the pre-trained baseline
