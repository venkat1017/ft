QLoRA: 4-Bit
Quantization

⏱ 75 min🔢 Quantization Theory💾 Memory Math

QLoRA fits a 7B model into 10GB by quantizing weights to 4-bit integers. But naively quantizing destroys precision - QLoRA uses NF4 (NormalFloat4), a data type designed specifically for the normal distribution of neural network weights. Understanding why this works is what separates engineers who can debug quantization from those who cargo-cult configs.

The Memory Math

Before understanding how QLoRA saves memory, understand what takes memory. A 7B parameter model in different precisions:

ComponentFP32 (4B)FP16/BF16 (2B)INT8 (1B)NF4 (0.5B)
7B weights28 GB14 GB7 GB3.5 GB
+ Optimizer states (Adam)+56 GB+28 GBN/A (frozen)N/A (frozen)
+ Gradients+28 GB+14 GBN/A (frozen)N/A (frozen)
+ Activations (batch=1)+2 GB+1 GB+1 GB+1 GB
Total (7B model)114 GB57 GB~8 GB~5-6 GB

QLoRA keeps the base model in NF4 (frozen) and only trains tiny LoRA adapters in BF16. The base model has no optimizer states or gradients - they're frozen. Only the adapters (~50MB) need full precision training states. This is how a 7B model fits in 10GB.

NF4: Why Not INT4?

Standard INT4 maps values uniformly: 0,1,2,...,15 → -8,-7,-6,...,7. But neural network weights follow a normal (Gaussian) distribution - most weights are near zero, few are large. A uniform quantization grid wastes precision on values that rarely occur.

NF4 uses non-uniform quantization levels derived from the normal distribution's quantile function. This means more grid points near zero (where most weights live) and fewer at the extremes. Same 4 bits, dramatically less quantization error.

nf4_intuition.py - why NF4 beats INT4
INTUITION
import torch
import scipy.stats as stats
import numpy as np

# INT4 quantization levels (uniform)
int4_levels = np.linspace(-1, 1, 16)
# [-1.0, -0.867, -0.733, -0.6, ..., 0.6, 0.733, 0.867, 1.0]

# NF4 levels (from normal distribution quantiles)
# Placed at quantiles of N(0,1) so each bucket has equal probability mass
quantiles = np.linspace(1/32, 31/32, 16)
nf4_levels = stats.norm.ppf(quantiles)
nf4_levels /= np.max(np.abs(nf4_levels))  # normalize to [-1, 1]
# [-1.0, -0.69, -0.52, -0.39, -0.28, -0.18, -0.09, 0.0,
#   0.09,  0.18,  0.28,  0.39,  0.52,  0.69,  1.0]
# ← more levels near zero (dense) fewer at extremes

# Quantization error comparison on realistic weight tensor
weights = torch.randn(1000)  # normal distribution of weights
weights_np = weights.numpy()
weights_norm = weights_np / np.max(np.abs(weights_np))

def quantize(w, levels):
    idx = np.argmin(np.abs(w[:, None] - levels[None, :]), axis=1)
    return levels[idx]

err_int4 = np.mean((weights_norm - quantize(weights_norm, int4_levels))**2)
err_nf4  = np.mean((weights_norm - quantize(weights_norm, nf4_levels))**2)
print(f"INT4 MSE: {err_int4:.6f}")  # ~0.0031
print(f"NF4  MSE: {err_nf4:.6f}")  # ~0.0019 - ~40% less error!

Double Quantization & Block-wise Quantization

QLoRA applies two more tricks beyond NF4. Block-wise quantization: instead of one scale factor for the entire weight matrix, use a separate scale per small block (64 weights). This dramatically reduces the impact of outlier weights. Double quantization: the scale factors themselves are quantized (from FP32 to 8-bit), saving another ~0.5GB on a 7B model.

bnb_config.py - loading a 4-bit model with bitsandbytes
PRODUCTION CODE
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                       # enable 4-bit loading
    bnb_4bit_quant_type="nf4",               # NF4 data type (not INT4!)
    bnb_4bit_compute_dtype=torch.bfloat16,    # computation in BF16
    bnb_4bit_use_double_quant=True,           # double-quantize the scales
)

# This downloads ~4GB for a 7B model (vs 14GB in FP16)
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B-model-Instruct",
    quantization_config=bnb_config,
    device_map="auto",           # auto-place on GPU(s)
    torch_dtype=torch.bfloat16,  # for non-quantized layers (norms, etc)
    trust_remote_code=True,
)

# Check VRAM usage
print(f"GPU mem allocated: {torch.cuda.memory_allocated()/1e9:.1f} GB")
# Expected: ~5.2 GB for Llama-3-8B-model (T4 has 15GB - we have room for adapters)

# Prepare for k-bit training (important!)
# This: casts layer norms to FP32, enables gradient checkpointing hooks
from peft import prepare_model_for_kbit_training
model = prepare_model_for_kbit_training(model)

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B-model-Instruct")
tokenizer.pad_token = tokenizer.eos_token  # Llama-3 has no pad token by default
✓
prepare_model_for_kbit_training() is non-optional. It casts the layer norms (RMSNorm weights) from BF16 to FP32 - critical because norms need full precision even when weights are quantized. Skipping it causes NaN losses and very poor model quality.

What You Now Know

  • 7B model memory: 28GB (FP32) → 14GB (FP16) → 5-6GB (NF4 + QLoRA adapters)
  • NF4 uses quantile-based levels matching the normal distribution of neural weights
  • Block-wise quantization: one scale per 64 weights (not per matrix)
  • Double quantization: scale factors are themselves quantized to 8-bit
  • prepare_model_for_kbit_training(): mandatory before adding LoRA adapters
